Http-протокл работает по своей природе как запрос-ответ. Но иногда надо хранить информацию между запросами. Для этого разработан механизм сессии - позволяющий сохранять информацию между запросами одного пользователя.

Есть три способа отслеживания сессии:
1) cookies
2) переопределяемый URL (используется response.encodeURL() для каждой ссылки, который вставляет идентификатор сессии в каждый URL.)
3) спрятанные поля HTML-страницы (в виде скрытых полей в форме)


На сервере приложений создается объект, который хранит различную информацию - он собственно сессия и называется. Сессии присваивается идентификатор. Для того, чтобы сервлет понял, с кем работает, он должен передать идентификатор сессии клиенту, чтобы клиент при следующих запросах мог передать этот идентификатор обратно серверу. При следующем коннекте клиент передает серверу через куки, или через url этот идентификатор, и сервлет получает по id сессии нужный ему объект.

Алгоритм:
- Создается сессия (объект) на сервере, ей присваивается идентификатор.
- идентификатор передается на клиента.
- клиент делает новый запрос, передавая id сессии
- сервер по id ссессии получает нужную ему сессию
Если куки отключены, то клиенту негде хранить id сессии и он вынужден передавать его через url.

Сервлет - это тоже является Java-приложением, которое управляется контейнером сервлетов (тоесть, работает под управлением сервера приложений).
По сути это класс-наследник типа 'HttpServlet', он имеет такие методы-слушатели:
- service
- doGet
- doPost
- doPut
- doDelete
- doFilter

Обращаться к методам слушателям сервлета можно через сервер приложения по HTTP-протоколу.
Сервлет имеет скоупы, такие как:
- page
- request
- session
- application
которые определяют время жизни данных клиентского запроса в сервлете
(И управляется такой сервлет сервером приложения)
